개요

딱 두 가지만

레포는 2개다 — MGMT research-dashboard(문서·킷·대시보드) + CODE(프로젝트별 구현). 누가 무엇을 하는가와 언제 무엇을 하는가, 이 둘만 잡으면 나머지는 순서대로 따라가면 된다.

나 나

킥오프·민감값 제공 · HUMAN GATE 판정 · PR merge. 정본은 issue 코멘트뿐.

세션 A 세션 A

회고·리포트·차기 P{k}.md·이슈. 웹 pro 또는 Claude 세션. 쓰기는 branch+PR로만.

§1

최초 1회 — 시스템 준비

처음 한 번, 약 20분. 전부 나가 직접 한다. 아래 넷을 순서대로. 레포·프로젝트 셋업은 프로젝트마다(§ 새 프로젝트)에서 하고, 여기서는 내 계정·워크스테이션·폰만 준비한다.

  1. 1

    계정 규율 — agent PAT 발급

    사람 = jiminc77, 에이전트 = jiminc77-agent로 나눈다. gjc·세션 A·gate-watcher 등 자동화 주체는 agent PAT로만 활동한다 — 이 분리가 유지되는 한 ### GATE VERDICT를 비-사람이 흉내 내면 verdict-guard가 자동 무효·원복한다(실효 조건).

    shell — agent PAT를 파일로 저장 (secret, 커밋 금지)bash나
    # github.com/settings/tokens 에서 jiminc77-agent 로 발급
# (두 레포 Contents RW + Issues RW, 스코프 최소)
mkdir -p ~/.config/gh-agent
printf '%s\n' "<PASTE_AGENT_PAT>" > ~/.config/gh-agent/token
chmod 600 ~/.config/gh-agent/token
# 게시 래퍼 safe-comment.sh 의 계정 게이트가 요구하는 기대 로그인
echo 'export AGENT_LOGIN=jiminc77-agent' >> ~/.bashrc

    확인: GH_TOKEN="$(cat ~/.config/gh-agent/token)" gh api user -q .login → jiminc77-agent. safe-comment.sh는 AGENT_LOGIN과 현재 gh 신원이 일치해야만 게시를 허용한다.

  2. 2

    워크스테이션 — gate-watcher + 경보 설치

    원격 워크스테이션에 systemd 데몬을 올린다. 이후 판정 코멘트를 감지하면 살아있는 gjc tmux에 "가서 읽어라" 신호만 전달한다(본문 주입 없음). 죽거나 멈추면 폰으로 경보하는 보조 유닛도 함께 건다.

    shell — 원격 워크스테이션bash나
    # 1) gate-watcher 데몬 (재부팅 자동 생존)
cp ~/Workspaces/research-dashboard/research-ops/gate-watcher/gate-watcher.service \
   ~/.config/systemd/user/
systemctl --user daemon-reload
systemctl --user enable --now gate-watcher
loginctl enable-linger $USER

# 2) 경보 유닛 3개 + OnFailure drop-in (죽으면 즉시 · 비활성이면 5분 내 ntfy)
cd ~/Workspaces/research-dashboard/research-ops/gate-watcher/alerts
cp gate-watcher-onfailure.service gate-watcher-liveness.service \
   gate-watcher-liveness.timer ~/.config/systemd/user/
systemctl --user edit gate-watcher   # [Unit] 아래 OnFailure=gate-watcher-onfailure.service
systemctl --user daemon-reload
systemctl --user enable --now gate-watcher-liveness.timer

    확인: systemctl --user status gate-watcher · 유닛 원본은 gate-watcher / alerts README.

  3. 3

    MGMT 브랜치 보호

    MGMT 게시는 branch+PR로만, pr-verify green 후 나가 merge한다(직접 main push 금지). MGMT Settings → Branches → Add rule → main에 Require a pull request + Require status checks에서 required check로 pr-verify 지정. (CODE 레포 브랜치 보호는 gjc 직접 push 의존이 사라지는 P1 종료 시 도입 — § 단계 루프 참고.)

    확인: MGMT에 아무 브랜치로 PR을 열면 pr-verify가 required로 걸리는지.

  4. 4

    폰 — ntfy 앱 설치

    게이트·킥오프·watcher 경보가 전부 ntfy로 온다. 폰(또는 웹)에 ntfy 앱만 깔아 둔다 — 프로젝트별 topic 구독은 프로젝트 착수 때 한다(§ 새 프로젝트).

    확인: curl -d "test" ntfy.sh/<아무_topic> → 그 topic을 구독한 폰에 뜨면 정상.

§2

새 프로젝트

프로젝트마다, 약 10분. 마법사로 지시서를 만들어 계획서 초안과 함께 새 세션 A에 붙여넣으면, 세션 A가 셋업(라벨·워크플로·설정·대시보드)부터 계획서 정형화·P0까지 수행한다.

1 · 마법사에 값 채우고 지시서 A 생성

필수 항목을 모두 채워주세요.

지시서 A — 세션 A 킥오프prompt 나세션 A

새 세션에 이걸 붙여넣을 때 계획서 초안을 첨부하세요.

2 · 세션 A가 자동으로 하는 일

지시서 A를 붙여넣으면 세션 A가 아래를 순서대로 수행한다. 게시는 전부 branch+PR, pr-verify green 후 내가 merge.

  • 라벨 상태기계 18종(양 레포)·phase 마일스톤(CODE 레포) 생성
  • 워크플로 4종(pr-verify · phase-transition · gate-notify · evidence-verify) + .github/immutables.txt를 CODE 레포로 복사·커밋
  • NTFY_TOPIC secret 등록 안내(gh secret set NTFY_TOPIC -R {OWNER}/{CODE_REPO})
  • MGMT projects/{slug}/project.yml 작성 — 민감값(ssh_host/workdir/모델)은 공란, human_login·agent_login은 실제 값으로 채움(safe-comment.sh·audit_actors.sh·gate-watcher가 읽는 계정 계약 정본) → branch+PR
  • 별도 대시보드 등록 없음 — project.yml이 유일한 레지스트리다. merge되면 dashboard-data Actions가 스캔해 dashboard/data.json을 굽는다(CONFIG.projects 하드코딩·titlePrefix 폐지)
  • 계획서 초안을 학회 수준으로 정형화 → 내 승인 후 P0.md 명세·dev 이슈 발행

3 · 사람 확인 세 가지

1 · secret 승인 나

세션 A가 안내하는 대로 CODE 레포에 NTFY_TOPIC secret을 등록·승인한다. 위치: CODE repo → Settings → Secrets and variables → Actions.

2 · 폰 topic 구독 나

폰 ntfy 앱에서 이 프로젝트의 새 topic을 구독한다(topic은 프로젝트마다 다름). 이 채널로 게이트·킥오프 푸시가 온다.

3 · 워크스테이션 전환 나

레포 쪽은 세션 A가 자동. 사람은 워크스테이션 넷만 만진다:

  1. ~/.config/gate-watcher/env의 NTFY_TOPIC만 새 topic으로 — 경보 유닛·타이머는 그대로
  2. watcher 설정을 새 프로젝트(CODE repo·ledger)로 바꾸고 systemctl --user restart gate-watcher
  3. 폰 ntfy 앱에서 새 topic 구독
  4. gjc 인증(GH_TOKEN, agent 계정)은 단계·프로젝트 경계에서만 교체 — 실행 중 전환 금지 (SESSION_B.md §1)

스크립트를 직접 실행하는 셸 셋업 경로(bootstrap_project.sh 등)는 research-ops/README.md 참조.

§3

단계 루프 — 단계마다 반복

세션 A가 P{k}.md·dev 이슈를 만든다 → 내가 gjc를 1회 부팅 → gjc가 자율 실행하며 증거·게이트를 게시 → 게이트는 내가 판정 → 단계 끝나면 킥오프 이슈가 자동 생성돼 다음 단계로.

⚠️ 보조 트랙 예외 (2026-07 신설, paper-sprint 등): phase 로드맵과 병행하는 시간-박스 트랙은 이 단계 루프를 따르지 않는다 — P{k}.md를 만들지 않고, 실행 명세는 델타 스펙(예: sprint_spec.md)으로 CODE 레포에 커밋한 뒤 커밋 SHA를 Decision 이슈에 핀(사전등록)한다. 이슈는 type:paper-sprint 라벨만(phase 라벨 금지), 게이트는 class:hard만 사용하며, 트랙 진행 중에는 해당 phase의 마지막 이슈(M6 sign-off류) close를 트랙 종료 게이트 이후로 유예한다(자동 킥오프 방지). 정본: research-ops/PROTOCOL.md "보조 트랙" 절 · Decision #35/#36.

1 · 흐름 한 눈에

P{k}.md · dev 이슈 세션 A→ gjc 부팅 나→ 자율 실행 · EVIDENCE gjc→ GATE VERDICT 나→ 킥오프 이슈 자동
dev 이슈는 항상 state:* 라벨 정확히 1개: ready→ running→ verify→ done · blocked-human blocked-tech

2 · 단계 시작 — 두 갈래

A · gjc 부팅 나

새 단계 실행 시작

이슈가 생겨도 gjc는 자동 시작되지 않는다. 원격 워크스테이션에서 1회 부팅하면 이후 자율 실행한다. 인증은 agent PAT를 주입한 뒤 tmux를 연다(단계 경계에서만 교체).

gjc 부팅 (원격 워크스테이션)bash나
ssh {SSH_HOST} && cd {WORKDIR} && git pull
tmux new -s gjc-P{k}
gjc ralplan --interactive "P{k}.md 명세로 실행 계획"
gjc ultragoal create-goals --brief-file P{k}.md
B · 다음 단계로 넘기기 나

단계 종료 → 자동 킥오프

마지막 phase:P{k} open 이슈를 close하면 phase-transition이 "[P{k+1}] 세션 A 킥오프" 이슈를 만들고 폰에 푸시한다. 그 이슈를 열고, 안의 두 옵션 중 하나로 세션 A를 실행한다:

(a) Claude/Cowork — 이슈 본문 프롬프트를 새 세션 A에 붙여넣기. (b) 웹 pro — 이슈의 완성본 프롬프트를 그대로 복붙(파라미터 치환 완료; 실패 시 계약 프롬프트 폴백). 산출물 4종을 PR로 올리면 pr-verify green 후 나가 merge.

gjc 수동 킥오프 상세 — 부팅 시퀀스(전환 이슈 없이 열 때)

tmux를 열기 전에 agent PAT를 주입한다. gjc는 jiminc77-agent 명의로 활동한다(verdict-guard 실효 조건). 단계마다 새 gjc 세션 1개를 띄운다(기존 세션 재사용 금지). 모델은 gjc 안에서 default/planner/architect=main · executor/critic=exec로 설정하고, ralplan 대화형 계획을 사람과 확정한 뒤에만 ultragoal로 진행한다. 전체 시퀀스·감독 규약은 SESSION_B.md 참조.

3 · 게이트 대응

ntfy 푸시가 오면 이슈를 열어 ### GATE VERDICT 코멘트를 남긴다. 판정은 오직 나, 사람 계정(jiminc77)으로만 — 게시 후 실행 세션 재개 전달은 gate-watcher가 자동으로 한다.

SOFT 자동

default·deadline 포함. deadline 지나면 gate-notify가 default 자동 채택. 다르게 가려면 그 전에 VERDICT로 override.

HARD 자동 리마인더 · 나 판정

무기한 대기, 6h마다 리마인더. go/no-go·임계값 변경·데이터 폐기·아키텍처 방향은 항상 hard.

### GATE VERDICT (내가 이슈에 남김)comment나
### GATE VERDICT
id: P1-M3-G1              # GATE REQUEST 의 id 그대로
choice: B
rationale: guard 여유 확인, 되돌릴 수 있음. M4로 진행.
follow-ups:
- 추가 지시 (없으면 생략)
# 게시하면 gate-notify가 blocked-human → ready 로 되돌리고 gjc가 집행

첫 줄이 정확히 ### GATE VERDICT여야 하고 choice: 줄이 반드시 있어야 한다. 사전등록 임계를 바꾸는 판정은 게이트 통과가 아니라 MGMT [Decision] 이슈를 거친다. 안 깨어나면 § 문제 해결.

4 · gjc의 증거·진행 게시

gjc가 게시하는 형식(참고 — 각 블록 상단 배지가 주체). 완료 신호는 EVIDENCE, 진행 보고는 PROGRESS 코멘트 1개를 편집한다.

### EVIDENCE (완료 신호)commentgjc
### EVIDENCE
goal-id: P1-M2
commits:
- 9f3a2b1        # origin/main 실재 SHA (CI가 재검증)
metrics:
  primary: {name: accuracy, value: 0.71,
            threshold: ">=0.65", pass: true}
  guard:   {name: success_rate, value: 0.06,
            random: 0.04, ok: true}
# CI VERIFIED ✅ 뜬 뒤에만 close
### PROGRESS (진행 — 코멘트 1개 편집)commentgjc
### PROGRESS
- [x] AC1 데이터 로더
- [ ] AC2 학습 루프  ← 진행 중
updated: 2026-07-06T09:12Z
# 새 댓글은 GATE REQUEST·EVIDENCE·GATE VERDICT 뿐
# 진행 보고는 이 댓글을 편집 (≥4h 간격)

guard 이상치면 primary가 PASS여도 정지. gjc는 ok:false를 보면 EVIDENCE 대신 GATE REQUEST(class:hard)로 전환한다. 3-strike(같은 문제 3회 실패)면 state:blocked-tech + HALT.

§4

문제 해결

실행 중 자주 막히는 세 가지. 커맨드는 그대로 실행하면 된다.

ntfy가 안 온다 나
shellbash나
gh secret list -R {OWNER}/{CODE_REPO} \
  | grep NTFY_TOPIC
curl -d "test" ntfy.sh/{topic}   # 폰에 뜨면 정상

curl 줄은 아무 로컬 셸에서 실행(topic 문자열만 알면 됨, 인증 불필요). 구독은 폰/웹 ntfy 앱에서 그 topic(프로젝트마다 다름). secret 위치: CODE repo → Settings → Secrets and variables → Actions → NTFY_TOPIC.

gjc가 안 깨어난다 나
shell — 원격 워크스테이션bash나
systemctl --user status gate-watcher
tail -20 ~/.local/state/gate-watcher/watcher.log
systemctl --user restart gate-watcher   # 필요 시

체크 순서: ① 판정 첫 줄이 정확히 ### GATE VERDICT이고 choice: 줄이 있는지 ② 이슈에 state:blocked-human 라벨이 붙었는지(없으면 수동 부착) ③ 판정 코멘트에 👀가 달렸는지(달렸으면 전달됨). 최후엔 tmux 수동 nudge.

watcher가 죽었다 · 경보 나

경보 유닛이 걸려 있으면 죽으면(OnFailure) 즉시, 비활성이면(liveness) 5분 내 ntfy로 알려준다. 재기동·경보 설치는 § 최초 1회의 gate-watcher 카드와 동일 커맨드. 유닛 원본: gate-watcher/alerts.

계정 규율(#22): 에이전트(gjc·세션 A·gate-watcher)는 jiminc77-agent PAT로 활동한다 — 사람 PAT를 에이전트에 넣지 않아야 verdict-guard가 실효한다. 게시 전 scripts/safe-comment.sh가 gh api user == AGENT_LOGIN을 강제하고(불일치 거부), 사후 감사는 scripts/audit_actors.sh <owner/repo> <issue>. gjc 인증 교체는 단계 경계에서만. 유출 의심 시 즉시 revoke.
