30초 개념 요약

역할 4개, 규칙 1개

사람은 판정만 한다. 나머지는 오케스트레이터가 정형화하고, 실행 에이전트가 코드로 증명하고, 자동화가 밀어준다. 상태는 기억이 아니라 GitHub 라벨에 있다.

세션 A — 오케스트레이터

계획 정형화 · milestone · 단계 명세 · dev 이슈 발행 · 검증 설계. 게이트 땐 판단 보조(코파일럿).

gjc 실행 에이전트

코드 구현, 이슈 open/close, evidence·gate 코멘트. owner 계정으로 gh CLI 사용. 부팅·감독은 세션 B가 담당.

사람 Human Gate

HUMAN GATE 판정 · gjc steer 승인. 그 외에는 개입하지 않는다.

자동화

GitHub Actions + ntfy 푸시 + 라이브 대시보드. 게이트 감지·증거 재검증·리마인더.

Docs

= 지식. 무엇을·왜 하는가. projects/{name}/에 산다.

Issues

= 상태. 지금 어디까지 왔는가. dev 이슈 1개 = goal 1개.

라벨

= 상태기계. dev 이슈는 항상 state:* 라벨 1개.

Code repo

= 증거. 커밋 SHA로 재현되는 결과가 진실이다.

State Machine

라벨 상태기계

모든 dev 이슈는 아래 다섯 상태 중 정확히 하나를 가진다. 막히면 두 분기 라벨로 빠진다.

state:ready→ state:running→ state:verify→ state:done
분기: state:blocked-human HUMAN GATE 대기 (GATE REQUEST) · state:blocked-tech 기술 블로커 (3-strike)
시작점 두 가지

어디서 시작하는가?

두 흐름은 다르다. 프로젝트를 처음 만드는가(A · 1회) 아니면 이미 있는 프로젝트의 다음 단계를 여는가(B · 매 단계). 헷갈리면 여기서 갈라져 내려가세요.

A · 프로젝트당 1회

완전히 새 연구 프로젝트를 시작할 때

CODE 레포 준비 → 라벨·마일스톤 부트스트랩 → 강제 인프라(워크플로+ntfy) → 대시보드 등록 → 계획서 정형화. 한 번만 한다.

A 셋업 절차 보기 →
B · 매 단계

다음 단계 세션을 시작할 때

기억이 아니라 GitHub 라벨에서 상태를 복원하는 새 오케스트레이터 세션. 킥오프 프롬프트 → self-check → 단계 명세 → dev 이슈 → gjc 위임.

B 단계 절차 보기 →
A · 셋업

완전히 새 연구 프로젝트를 시작할 때

커맨드는 없다. 폼 채우기 → 지시서 복사 → 새 세션에 계획서 초안 첨부해 붙여넣기 → 세션 A가 셋업부터 다 한다.

필수 항목을 모두 채워주세요.

지시서 A — 세션 A 킥오프prompt

새 세션에 이걸 붙여넣을 때 계획서 초안을 첨부하세요.

수동으로 직접 하려면 (참고)
  1. 1

    구현(CODE) 레포 준비

    새 repo를 만들거나 기존 repo를 지정합니다. 아래에서 <CODE_REPO> 자리에 이 레포 이름이 들어갑니다.

    shellbash
    # 새로 만들 경우
gh repo create {OWNER}/{CODE_REPO} --private --clone
# 또는 기존 레포를 그대로 CODE 레포로 지정
  2. 2

    라벨 · 마일스톤 부트스트랩

    MGMT 레포의 재사용 키트로 CODE 레포에 라벨 18종(state 상태기계 + 보조)과 phase 마일스톤을 한 번에 생성합니다.

    bootstrap_project.shbash
    bash research-ops/scripts/bootstrap_project.sh \
  {OWNER} {MGMT_REPO} {CODE_REPO} {PROJECT} \
  "P0:scaffold,P1:baseline,P2:probing-decision,P3:report"

    인자 순서는 <owner> <mgmtRepo> <CODE_REPO> <PROJECT> "P0:…,P1:…". 라벨만 다시 깔고 마일스톤은 건드리지 않으려면 --labels-only 플래그를 붙입니다 — 이미 굴러가는 프로젝트에 상태기계 라벨을 재보정할 때 안전합니다.

  3. 3

    강제 인프라: 워크플로 2개 + ntfy

    두 GitHub Actions 워크플로를 CODE 레포 .github/workflows/에 커밋합니다. gate-notify.yml은 blocked-human 라벨/GATE REQUEST 코멘트를 감지해 ntfy 푸시·6h 리마인더·soft 시한 자동 채택을 하고, evidence-verify.yml은 EVIDENCE 코멘트의 커밋 SHA를 checkout해 pytest를 재실행하고 ✅/❌ 라벨을 답니다.

    shellbash
    # 1) 워크플로 커밋
git add .github/workflows/gate-notify.yml .github/workflows/evidence-verify.yml
git commit -m "ci: gate-notify + evidence-verify" && git push

# 2) ntfy topic 시크릿 등록 (프로젝트마다 고유 topic 권장)
gh secret set NTFY_TOPIC -R {OWNER}/{CODE_REPO}
#    값 형식: {project}-gates-{랜덤}

# 3) 폰 ntfy 앱에서 그 topic 구독
#    ntfy.sh/{topic}  ← Subscribe

    topic은 프로젝트마다 다릅니다. 새 프로젝트를 만들 때마다 새 topic을 정하고, 폰에서 그 topic을 새로 구독해야 푸시가 옵니다.

  4. 4

    대시보드 등록

    MGMT 레포의 dashboard/index.html 안 CONFIG.projects 배열에 새 프로젝트 항목을 추가하고 커밋합니다. MGMT 레포는 공용이므로, milestone 제목에는 [PROJECT] 접두사가 붙습니다.

    dashboard/index.html — CONFIG.projectsjs
    projects: [
  // 기존 프로젝트 항목 …
  {
    name: "{PROJECT}",
    owner: "{OWNER}",
    mgmtRepo: "{MGMT_REPO}",
    codeRepo: "{CODE_REPO}",
    titlePrefix: "[{PROJECT}] "
  }
]

    신규 프로젝트는 공용 MGMT 레포에서 titlePrefix(예: [{PROJECT}] )로 구분합니다.

  5. 5

    계획서 정형화 세션

    오케스트레이터 세션에 "세션 실행 지시서"(계획서 초안 + CODE repo)를 전달합니다. ORCHESTRATOR.md STEP 1–2가 plan_refinement.md 절차로 초안을 학회 수준으로 정형화하고 projects/{project}/research/에 .md + .html로 남깁니다. 마지막에 HUMAN 승인으로 확정합니다.

    킥오프 — 새 프로젝트 계획서 정형화prompt
    새 연구 프로젝트의 계획서 정형화를 맡긴다. research-dashboard의
research-ops/ORCHESTRATOR.md 와 PROTOCOL.md, templates/ 를 읽고 그대로 따른다.
STEP 0 self-check → STEP 1–2(계획서 정형화)까지 수행한다.
- 구현 레포: {OWNER}/{CODE_REPO} · 관리 레포: {OWNER}/{MGMT_REPO}
- 첨부: 계획서 초안(아래). templates/plan_refinement.md 절차로 학회 수준까지
  정형화하고, research_plan_template.md 구조에 맞춰 projects/{project}/research/ 에
  .md + .html(plan_html_guide.md 준수) 두 산출물로 커밋한다.
- 사전등록: 각 phase의 primary+guard 지표와 HUMAN GATE 지점을 계획서에 고정한다.
- 라벨·마일스톤 부트스트랩은 이미 끝났다고 가정(아니면 STEP 0에서 지적).
- 정형화 결과는 HUMAN 승인 전까지 확정하지 말 것. 승인 요청으로 종료.

[여기에 계획서 초안 붙여넣기]
  6. 6

    이후 단계 루프는 B로

    셋업이 끝나면 실제 연구는 단계 단위로 진행됩니다. P0, P1, P2… 각 단계는 아래 B. 다음 단계 세션 절차를 반복하세요.

B · 매 단계

다음 단계 세션을 시작할 때

단계마다 세션 두 개를 연다 — 세션 A(메인 오케스트레이터)와 세션 B(gjc 감독). A가 문서·이슈·게이트를 굴리고, B는 워크스테이션에서 gjc를 부팅·감시한다. 둘 다 기억이 아니라 projects/{PROJECT}.yml과 GitHub 라벨에서 상태를 복원한다.

세션 A

메인 오케스트레이터 · 게이트 코파일럿

전환 이슈 지시서로 부팅 → projects/{PROJECT}.yml로 설정 복원 → 이전 단계 회고 → P{k+1}.md·dev 이슈 발행 → 마지막에 세션 B 지시서 출력. 게이트 땐 상황 요약·선택지 분석으로 판단을 돕는다(판정은 사람).

세션 B

gjc 부팅 · 감독

세션 A가 준 지시서를 붙여넣으면 ssh → tmux → gjc 새 세션(모델 구성) → ralplan → ultragoal로 부팅 후 감독한다. 응답 대기 우선 — steer 전에는 사용자에게 먼저 문의한다.

핵심: 너는 커맨드를 타이핑하지 않는다. 전환 이슈·세션 B 지시서를 복사해 붙여넣기만 하면 된다. 상태의 단일 진실원은 GitHub 라벨과 projects/{PROJECT}.yml이다.

  1. 1

    전환 이슈 자동 생성 → ntfy 푸시

    직전 단계의 마지막 dev 이슈가 닫히면 phase-transition 워크플로가 "[P{k+1}] 세션 A 킥오프" 이슈를 자동 생성하고 ntfy로 폰에 푸시합니다. 당신이 할 일은 그 이슈를 여는 것뿐.

  2. 2

    세션 A 부팅 — 전환 이슈 지시서 붙여넣기

    그 이슈 본문의 지시서를 복사해 새 세션 A에 붙여넣습니다. 세션 A는 projects/{PROJECT}.yml로 설정을 복원하고, 이전 단계 회고 후 P{k+1}.md를 쓰고 setup_phase.sh로 dev 이슈를 발행합니다.

  3. 3

    세션 A가 세션 B 지시서 출력

    이슈 발행이 끝나면 세션 A가 마지막에 세션 B 실행 지시서(SESSION_B.md 템플릿에 값 채움)를 복사 블록으로 출력하고 정지합니다.

  4. 4

    세션 B 부팅 → gjc 감독

    그 지시서를 새 세션 B에 붙여넣으면 B가 ssh → tmux → gjc 새 세션(모델 구성) → ralplan → ultragoal 부팅 후 감독을 시작합니다. 이후 게이트가 뜨면 아래 게이트 대응으로.

수동 킥오프 (전환 이슈를 안 쓸 때)
  1. 1

    새 세션에 킥오프 프롬프트 붙여넣기

    아래 단계 킥오프 프롬프트를 새 오케스트레이터 세션 첫 메시지로 그대로 붙여넣습니다. 단계 번호와 CODE/MGMT 레포만 바꾸면 다른 단계에도 재사용됩니다.

  2. 2

    STEP 0 self-check

    오케스트레이터가 착수 전 스스로 점검합니다.

    STEP 0 self-checkcheck
    # 오케스트레이터가 자동 확인하는 항목
gh auth status                       # ① gh 인증 (owner 계정)
gh label list -R {OWNER}/{CODE_REPO} # ② state:* 라벨 존재 확인
ls research-ops/templates PROTOCOL.md # ③ 템플릿·PROTOCOL 존재
gh repo view {OWNER}/{CODE_REPO}     # ④ CODE repo 접근 가능
  3. 3

    단계 명세 작성 — P{k}.md

    phase_spec.md 뼈대로 단계 명세를 씁니다. @goal 블록으로 M0..Mj를 정의하고, 각 Exit에 primary + guard 지표를 사전등록하며, HUMAN GATE에는 class(hard/soft)를 지정합니다. 완성되면 CODE repo에 push.

    P2.md — @goal 블록 (phase_spec.md 뼈대)markdown
    @goal M2 probe-policy
  summary: 관측 예산 하에서 probe 정책 학습
  exit.primary: decision-AUC ≥ 0.82        # 사전등록
  exit.guard:   probe-budget ≤ 1.0×, FP-rate ≤ 0.05
  artifact: code-repo commit + pytest tests/test_probe.py

@goal M3 human-gate
  gate.class: hard                          # 무기한 대기
  gate.ask:   probe 정책 배포 승인 여부
  4. 4

    dev 이슈 생성 — setup_phase.sh

    스크립트가 P{k}.md의 @goal 블록을 파싱해 dev 이슈를 만들고, 라벨·마일스톤을 붙이고, P{k}.md의 Goal↔Issue Map에 생성된 이슈 번호를 자동 backfill합니다.

    setup_phase.shbash
    bash research-ops/scripts/setup_phase.sh P2 projects/{project}/phases/P2.md
# → dev 이슈 생성 + state:ready 라벨 + P2 마일스톤 + 번호 backfill
  5. 5

    gjc 위임 → PROTOCOL 준수

    실행 에이전트 gjc에게 goal을 위임합니다. gjc는 계약(PROTOCOL.md)을 그대로 따릅니다.

    착수

    이슈를 state:running으로 전환.

    게이트

    GATE REQUEST 코멘트 + state:blocked-human.

    완료

    EVIDENCE 코멘트 → CI ✅ 뜬 뒤에만 close.

    진행

    PROGRESS 코멘트를 편집해 상태 갱신.

  6. 6

    게이트·종료 신호

    게이트가 뜨면 사람이 GATE VERDICT 코멘트를 남긴다 → 다음 goal로. 단계 전체가 끝나면 HUMAN sign-off → 다음 단계(P{k+1})로 넘어간다. 아래 게이트 대응 참고.

복사해서 바로 쓰는 프롬프트

단계 킥오프 프롬프트 (P{k})

다음 단계 오케스트레이터 세션의 첫 메시지. 아래를 그대로 복사해 붙여넣으세요. (단계 번호 P{k}와 참조 문서 경로만 교체.)

단계 오케스트레이션 킥오프 (P{k})prompt 
새 단계 오케스트레이션을 맡긴다. {MGMT_REPO}의 research-ops/ORCHESTRATOR.md 와 PROTOCOL.md, templates/ 를 읽고 그대로 따른다. STEP 0 self-check부터 시작(라벨·마일스톤은 이미 생성돼 있음).
- 구현 레포: {OWNER}/{CODE_REPO} · 관리 레포: {OWNER}/{MGMT_REPO}
- 단계: P{k} ({사전 등록 기준 문서가 있으면 경로 명시})
- P{k}.md 작성: templates/phase_spec.md 뼈대. @goal 블록마다 Exit에 primary+guard 지표 사전등록, HUMAN GATE는 class(hard/soft) 지정. 완료 후 CODE repo에 push.
- dev 이슈 생성: bash research-ops/scripts/setup_phase.sh P{k} <P{k}.md 경로> (라벨·마일스톤·번호 backfill 자동)
- gjc 위임 시 PROTOCOL 준수 명시: 착수 state:running, 게이트는 GATE REQUEST+state:blocked-human, 완료는 EVIDENCE(primary+guard) 후 CI VERIFIED ✅ 기다렸다 close, 진행은 PROGRESS 코멘트 편집. guard 이상치면 primary PASS여도 정지.

마지막 줄이 핵심 안전장치입니다 — guard 지표가 이상치면 primary가 PASS여도 정지합니다. 지표는 사전등록된 값으로만 판정하세요.

사람이 개입하는 유일한 순간

게이트가 뜨면?

ntfy 푸시 도착 → 이슈 열기 → GATE VERDICT 코멘트. 순서는 늘 같다. 게이트에는 두 종류가 있고 처리 방식이 다르다.

SOFT 게이트

default + deadline

기본값(default)과 시한(deadline)이 함께 온다. 시한이 지나면 자동으로 default를 채택하고 진행한다. 다르게 가려면 시한 전에 override 판정만 남기면 된다.

HARD 게이트

무기한 대기

사람이 답할 때까지 무기한 정지한다. gate-notify.yml이 6h마다 리마인더를 다시 밀어준다. 배포·되돌릴 수 없는 결정은 항상 hard.

GATE REQUEST (gjc가 남김)comment
GATE REQUEST
class: hard
ask: probe 정책을 배포에 반영할까요?
context: decision-AUC 0.84 (≥0.82 PASS),
         probe-budget 0.9× (guard PASS)
# soft라면 아래가 추가됨:
# default: adopt · deadline: 2026-07-06T18:00Z
GATE VERDICT (사람이 남김)comment
GATE VERDICT
decision: approve
note: guard 여유 확인. M3로 진행.
# 코멘트를 남기면 gate-notify가
# blocked-human 라벨을 해제하고 다음 goal로.

close 전 반드시 CI ✅. EVIDENCE 코멘트의 커밋 SHA를 evidence-verify.yml이 checkout → pytest 재실행 → ✅/❌ 라벨을 답니다. ✅가 뜨기 전에는 이슈를 닫지 마세요 — 재현되지 않는 결과는 증거가 아닙니다.

Q6 · 알림

ntfy 확인

게이트 푸시가 안 오면 십중팔구 topic 구독 문제다. topic은 프로젝트마다 다르며, CODE repo의 NTFY_TOPIC 시크릿 값과 정확히 같아야 한다.

1 · topic 확인

이 프로젝트의 topic = CODE repo의 NTFY_TOPIC 시크릿 값. 시크릿 존재 여부는 아래로 확인(값 자체는 표시되지 않음).

2 · 구독 확인

폰 ntfy 앱에서 그 topic이 구독 목록에 있는지 확인. 없으면 ntfy.sh/<topic>을 Subscribe.

3 · 테스트

한 줄로 즉시 테스트 푸시. 폰에 뜨면 파이프라인 정상.

ntfy 점검 & 테스트bash
# 시크릿이 등록돼 있는지 (값은 마스킹됨)
gh secret list -R {OWNER}/{CODE_REPO} | grep NTFY_TOPIC

# 테스트 푸시 — 폰 ntfy 앱에 "test"가 떠야 정상
curl -d "test" ntfy.sh/{topic}

프로젝트마다 새 topic을 새로 구독해야 한다. A-3에서 정한 <project>-gates-<랜덤> topic은 그 프로젝트 전용이다. 새 프로젝트를 시작했는데 알림이 안 오면, 폰에서 그 프로젝트의 topic을 구독했는지부터 확인하세요.

마무리

체크리스트

세션 전후로 훑어보세요. 왼쪽은 프로젝트당 1회, 오른쪽은 매 단계.

A · 셋업 (1회)
  • 셋업 위저드 폼 채우기 — OWNER·CODE_REPO·PHASES·SSH_HOST·WORKDIR·NTFY_TOPIC
  • 지시서 A 생성 → 복사
  • 새 세션에 계획서 초안 첨부해 붙여넣기 → 세션 A 실행
  • 세션 A가 안내한 NTFY_TOPIC 시크릿 등록 완료 확인
  • 폰 ntfy 앱에서 그 topic 구독 확인
B · 단계 (매번)
  • "[P{k+1}] 세션 A 킥오프" 전환 이슈 지시서 복사
  • 새 세션 A에 붙여넣기 → 회고·P{k+1}.md·이슈 발행
  • 세션 A가 출력한 세션 B 지시서 복사
  • 새 세션 B에 붙여넣기 → ssh·tmux·gjc 부팅·감독
  • 게이트 대응 — GATE VERDICT · steer 승인 · 단계 종료 sign-off

토큰 주의: gjc는 owner 계정의 fine-grained PAT로 gh를 씁니다. 필요한 repo·권한(issues·contents·actions)에만 스코프를 좁혀 발급하고, 세션이 끝나거나 유출이 의심되면 GitHub 설정에서 즉시 폐기(revoke)하세요.
