1

새 프로젝트 시작하기

나 레포 2개 준비 → 폼 채우기 → 지시서 복사 → 계획서 초안 첨부해 새 세션 A에 붙여넣기. 세션 A가 셋업(라벨·워크플로·설정·대시보드)부터 계획서 정형화·P0까지 수행합니다.

셋업 순서 (프로젝트당 1회)
  • 나 레포 2개 — MGMT(research-dashboard, 공용) + CODE repo(없으면 세션 A가 생성)
  • 나 아래 위저드에 값 채우고 지시서 A 생성 → 복사
  • 나 새 세션 A에 계획서 초안 첨부해 붙여넣기 → 세션 A가 STEP 0 셋업 실행
  • 나 세션 A가 안내하는 NTFY_TOPIC secret 등록 + 폰 ntfy 앱에서 그 topic 구독
  • 나 계획서 HUMAN 승인 → 세션 A가 P0.md 작성·dev 이슈 발행

세션 A가 STEP 0에서 자동 실행: bootstrap_project.sh(라벨 18종·마일스톤) → CODE repo .github/workflows/에 pr-verify · phase-transition · gate-notify · evidence-verify 4종 + .github/immutables.txt 복사 → projects/{name}/ + project.yml 생성 → 대시보드 CONFIG.projects에 항목 추가(docsBase 포함).

세션 A나 인증 — 실행 주체로 갈린다. 세션 A(위저드 지시서를 받는 LLM 세션·샌드박스)는 gh가 없다 → 내가 준 fine-grained PAT(두 레포 Contents RW + Issues RW)로 REST API를 쓴다. 아래 gh auth status류 커맨드는 내가 로컬 셸에서 직접 부트스트랩할 때만(전제: gh 설치 + gh auth login 완료).

필수 항목을 모두 채워주세요.

지시서 A — 세션 A 킥오프prompt 나세션 A

새 세션에 이걸 붙여넣을 때 계획서 초안을 첨부하세요.

커맨드로 직접 부트스트랩 (참고)
shell — 라벨·마일스톤 + 워크플로 4종bash나
# 0) 전제: gh 로그인 (두 레포 Contents RW + Issues RW)
gh auth status

# 1) 라벨 18종(양 레포) + CODE phase 마일스톤 — 멱등 (--labels-only 로 라벨만 재보정)
bash research-ops/scripts/bootstrap_project.sh \
  {OWNER} research-dashboard {CODE_REPO} {PROJECT} \
  "P0:환경파일럿,P1:베이스라인,P2:제안기법,P3:평가"

# 2) 워크플로 4종 + 불변값 매니페스트를 CODE repo 로 복사·커밋
cp research-ops/workflows/{pr-verify,phase-transition,gate-notify,evidence-verify}.yml \
   {CODE_REPO_LOCAL}/.github/workflows/
cp .github/immutables.txt {CODE_REPO_LOCAL}/.github/immutables.txt
git -C {CODE_REPO_LOCAL} add -A \
  && git -C {CODE_REPO_LOCAL} commit -m "ci: research-ops workflows" \
  && git -C {CODE_REPO_LOCAL} push

# 3) ntfy topic secret (프로젝트마다 고유 topic) → 폰 ntfy 앱에서 그 topic 구독
gh secret set NTFY_TOPIC -R {OWNER}/{CODE_REPO}   # 값: {project}-gates-{랜덤}

이후 projects/{name}/project.yml(_template.yml)과 dashboard/index.html CONFIG.projects 항목(docsBase 포함)을 추가하고, 계획서 정형화 세션으로 P0.md까지 진행합니다. 위 위저드가 이 전부를 세션 A에 위임하는 지시서를 만들어 줍니다.

단계(Phase) 실행하기

P{k}.md와 dev 이슈는 세션 A(웹 pro)가 GitHub 앱으로 직접 만듭니다(§4 흐름). 이슈가 생겨도 gjc는 자동 시작되지 않습니다 — 나가 원격 워크스테이션에서 1회 부팅해야 실행이 시작되고, 이후 gjc가 자율 실행합니다.

dev 이슈 수동 발행 (대안)bash나

3

게이트 판정하기

자동 ntfy 푸시 도착 → 나 이슈 열기 → ### GATE VERDICT 코멘트. 판정은 오직 나. 정본은 issue 코멘트뿐입니다.

SOFT 자동

default·deadline 포함. deadline 지나면 gate-notify가 default 자동 채택. 다르게 가려면 그 전에 VERDICT로 override.

HARD 자동 리마인더 · 나 판정

무기한 대기, 6h마다 리마인더. go/no-go·임계값 변경·데이터 폐기·아키텍처 방향은 항상 hard.

### GATE VERDICT (내가 이슈에 남김)comment나
### GATE VERDICT
id: P1-M3-G1              # GATE REQUEST 의 id 그대로
choice: B
rationale: guard 여유 확인, 되돌릴 수 있음. M4로 진행.
follow-ups:
- 추가 지시 (없으면 생략)
# 게시하면 gate-notify가 blocked-human → ready 로 되돌리고 gjc가 집행

사전등록 임계를 바꾸는 판정은 게이트 통과가 아니라 MGMT [Decision] 이슈를 거쳐야 합니다. ntfy가 안 오면 7 · 검증이 막을 때의 ntfy 점검을 보세요.

4

단계 넘어가기

마지막 phase:P{k}(현재 단계 라벨, 예: phase:P1) open 이슈를 close하면 킥오프 이슈가 자동 생성됩니다. 세션 A가 산출물 4종을 PR로 올리고, pr-verify green 후 내가 merge합니다.

자동 킥오프 → 나 복붙 → 세션 A PR → 자동 검증 → 나 merge
  1. 1

    자동킥오프 이슈 자동 생성 → ntfy 푸시

    phase-transition이 "[P{k+1}] 세션 A 킥오프" 이슈를 만들고 폰에 푸시합니다. 그 이슈를 엽니다.

  2. 2

    나복붙세션 A세션 A 실행 — 이슈 안의 두 옵션 중 하나

    (a) Claude/Cowork 경로 — 이슈 본문의 프롬프트를 새 세션 A에 붙여넣기.  (b) 웹 pro 경로 — GitHub 쓰기 앱이 붙은 웹 pro에 킥오프 이슈의 완성본 프롬프트(웹 pro 프롬프트 (완성본 — 그대로 복붙) 접기 블록)를 그대로 복붙합니다. 파라미터는 이미 전부 치환돼 있어 손댈 필요가 없습니다(치환 실패 시엔 계약 프롬프트 + 파라미터 블록으로 폴백).

  3. 3

    세션 A세션 A가 산출물 4종을 PR로

    P{k}_report.html · status.json · P{k+1}.md · 차기 dev 이슈들. 쓰기는 branch+PR로만(MGMT 1 PR + CODE 1 PR). PR 본문에 evidence 링크 + 불변값 이월 확인.

  4. 4

    자동pr-verify green나merge

    pr-verify 체크가 초록이 된 것을 확인하고 내가 merge합니다. 세션은 스스로 merge하지 않습니다. 실패하면 7 참고.

나 branch protection 켜는 시점 = P1 종료(sign-off). gjc의 직접 push 의존이 사라지면 CODE repo Settings → Branches → Add rule → main에 Require a pull request + Require status checks에서 pr-verify 지정. (P1 실행 중엔 켜지 않음 — gjc 직접 push가 막힘.)

ntfy 푸시가 안 온다 나
shellbash나
gh secret list -R {OWNER}/{CODE_REPO} \
  | grep NTFY_TOPIC
curl -d "test" ntfy.sh/{topic}   # 폰에 뜨면 정상

curl 줄은 아무 로컬 셸에서 실행 — topic 문자열만 알면 되고 인증 불필요. 구독은 폰/웹 ntfy 앱에서 그 topic을 구독(topic은 프로젝트마다 다름).
secret 위치: CODE repo → Settings → Secrets and variables → Actions → NTFY_TOPIC (워크플로가 여기서 읽어 푸시).

CI가 EVIDENCE를 ❌ 냈다 자동 나

evidence-verify가 커밋 SHA를 checkout해 pytest 재실행 → 실패 시 state:blocked-tech. SHA가 origin/main에 실재하고 테스트가 재현되는지 확인. 재현 안 되는 결과는 증거가 아님.

나 토큰: gjc는 owner 계정 fine-grained PAT(issues·contents·actions)로 gh를 씁니다. 스코프를 좁혀 발급하고, 끝나거나 유출 의심 시 GitHub 설정에서 즉시 revoke.
