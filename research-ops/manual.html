§2 · 상태 라벨

18종의 라벨이 곧 진실

별도 상태 파일은 없다. GitHub 라벨이 유일한 기계 판독 진실이며, 대시보드와 자동화는 이 라벨만 읽는다.

state — 6종 (이슈당 정확히 1개)

state 라벨색상의미다음 전이 주체
state:ready 회색 착수 대기 (이슈 생성 직후 기본, soft default 채택 후) gjc
state:running 파랑 gjc 구현 중 gjc
state:verify 보라 EVIDENCE 게시됨, CI 검증 대기 Actions
state:done 초록 CI VERIFIED ✅ 후 완료 · close —
state:blocked-human 빨강 HUMAN GATE 대기 (분기) 사람
state:blocked-tech 주황 기술 블로커 · 3-strike · CI 실패 (분기) gjc
!

단일 state 규칙: dev 이슈는 항상 state:* 라벨을 정확히 1개 보유한다. 0개거나 2개 이상이면 대시보드가 이상으로 표시한다.

type — 4종 · phase — 8종 (state와 병존)

type 라벨이슈 종류
type:dev구현 작업 이슈
type:gateHUMAN GATE 이슈
type:decisionMGMT 기준 변경 결정
type:milestoneMGMT 단계 마일스톤
phase 라벨의미
phase:P0 … phase:P7이슈가 소속된 단계 (P0–P7)
각 이슈는 소속 단계 phase 라벨 1개를 붙여 대시보드의 단계별 진행률로 집계된다.

§3 · GitHub 마일스톤

제목 접두어와 네이티브 마일스톤은 다르다

이슈 제목의 [P2-M1] 접두어는 사람 눈을 위한 관례일 뿐이다. 반면 GitHub 네이티브 milestone은 이슈들을 단계로 묶어 done/total 진행률을 GitHub이 자동으로 계산·필터해 주는 기능이다. 둘은 별개다 — 하나는 텍스트, 하나는 실제 집계.

setup_phase.sh가 dev 이슈를 생성할 때 해당 milestone에 자동으로 연결하므로, 손으로 이슈를 마일스톤에 넣을 필요가 없다. 그 결과 GitHub의 Milestones 화면에서 각 단계의 진행률이 바로 보인다.

P0·P1은 이미 지난 단계이므로 네이티브 milestone은 P2–P7만 생성돼 있다. 새 단계를 시작할 때 이미 만들어진 milestone을 재사용한다.

§4 · 케이스 A

완전히 새 연구 시작

아무것도 없는 상태에서 시작한다. 인프라 5단계를 먼저 세우고, 오케스트레이터 세션에 착수 프롬프트를 붙인다.

1

CODE 레포 생성

구현이 일어날 코드 레포 jiminc77/{CODE_REPO}를 GitHub에 만든다. 계획서 초안은 경로 또는 URL로 준비한다.

2

bootstrap_project.sh — 라벨 18종 + 마일스톤 생성 (1회)

양 레포에 state:*·type:*·phase:* 라벨과 phase별 GitHub 마일스톤을 멱등 일괄 생성한다. 프로젝트당 딱 한 번.

3

워크플로 2개 커밋 + NTFY_TOPIC secret

gate-notify.yml·evidence-verify.yml을 CODE 레포 .github/workflows/에 커밋하고, gh secret set NTFY_TOPIC으로 알림 토픽을 등록한다.

4

dashboard CONFIG.projects에 프로젝트 추가

MGMT 레포 dashboard/index.html의 CONFIG.projects 배열에 이 프로젝트 항목을 추가하고 커밋한다. 대시보드가 새 프로젝트를 인식한다.

5

오케스트레이터 세션에 착수 프롬프트

아래 PROMPT-NEW를 복사해 오케스트레이터 세션에 붙인다. STEP 0 self-check부터 인프라 구축과 P0 명세까지 스스로 진행한다.

★

인프라는 프로젝트당 딱 1회. 위 1–4단계(레포·라벨·마일스톤·워크플로·secret·대시보드 등록)는 프로젝트를 처음 세울 때만 한다. 이후 단계 진행은 케이스 B로 넘어간다.

복사해서 붙이는 프롬프트 · PROMPT-NEW

PROMPT-NEW — 새 연구 착수 
새 연구 프로젝트의 오케스트레이션을 맡긴다.
jiminc77/research-dashboard 의 research-ops/ORCHESTRATOR.md, research-ops/PROTOCOL.md, research-ops/templates/ 를 읽고 그대로 따른다. STEP 0 self-check부터 시작한다.

- 계획서 초안: {경로 또는 URL}
- 구현 레포: jiminc77/{CODE_REPO}
- 프로젝트명: {PROJECT}
- 실행 환경: SSH_HOST={..} / WORKDIR={..} / HARDWARE={..} / 도구={..}

인프라 미구축 상태면 먼저 수행:
1) bash research-ops/scripts/bootstrap_project.sh jiminc77 research-dashboard {CODE_REPO} {PROJECT} "P0:{이름},P1:{이름},..."
2) research-ops/workflows/*.yml 을 {CODE_REPO}/.github/workflows/ 에 커밋
3) gh secret set NTFY_TOPIC -R jiminc77/{CODE_REPO}   # 값: 기존 topic 재사용
4) dashboard/index.html 의 CONFIG.projects 에 {PROJECT} 항목 추가 후 커밋
이후 STEP 1→6: 계획서 보강·정형화(HUMAN 승인) → milestone → P0.md(사전등록 포함) → setup_phase.sh → gjc 위임.

§5 · 케이스 B

진행 중 프로젝트의 새 단계 (P2 예시)

인프라는 이미 서 있다. 이전 단계 종료를 검증하고, 바로 단계 명세부터 작성한다.

케이스 A와의 차이

항목케이스 A (새 연구)케이스 B (새 단계)
인프라 구축 레포·라벨·마일스톤·워크플로·대시보드 등록 스킵 — 이미 존재
시작 지점 계획서 보강부터 (STEP 1–6) P{k}.md 작성부터
선행 조건 없음 이전 단계 종료 검증 필수 — dev 이슈 전부 closed + HUMAN sign-off 존재

복사해서 붙이는 프롬프트 · PROMPT-PHASE

PROMPT-PHASE — 새 단계 (P2 예시) 
DGCC P2 단계의 오케스트레이션을 맡긴다.
jiminc77/research-dashboard 의 research-ops/ORCHESTRATOR.md 와 research-ops/PROTOCOL.md 를 읽고 그대로 따른다.

1) STEP 0 self-check
2) P1 종료 검증: phase:P1 dev 이슈 전부 closed, HUMAN sign-off 이슈에 GATE VERDICT 존재. 미완이면 정지하고 나에게 보고.
3) P2.md 작성 (jiminc77/DGCC 루트, P0.md/P1.md 형식 계승):
   - 입력: docs/research/DGCC_research_plan.md §10 의 P2 정의 + docs/reports/P2_probing_decision.md 의 사전 고정 임계 (수치 변경 금지)
   - @goal: M0..Mj 각 블록 Exit 에 primary+guard 지표와 판정 기준을 사전등록
   - HUMAN GATE goal 은 class(hard|soft) 지정, soft 는 default+deadline 포함
   - "실행 환경(고정)" 블록은 P1.md 승계
4) P2.md 요약을 나에게 보고하고 승인받은 뒤 push
5) bash research-ops/scripts/setup_phase.sh P2 {DGCC 로컬 경로}/P2.md   # 이슈·라벨·마일스톤 연결·번호 backfill 자동
6) gjc 위임 (PROTOCOL.md 준수 명시) → 완료 검증은 ORCHESTRATOR §4 체크리스트

§6 · 케이스 C

중단 세션 재개

세션이 끊겼다면 상태를 GitHub에서 복원한다. 라벨 쿼리 한 방으로 지금 사람을 기다리는지, 어디까지 왔는지 알아낸다.

복사해서 붙이는 프롬프트 · PROMPT-RESUME

PROMPT-RESUME — 세션 재개 
{PROJECT} 오케스트레이션을 재개한다.
research-ops/ORCHESTRATOR.md 와 PROTOCOL.md 를 읽은 뒤 상태를 복원한다:
- bash research-ops/scripts/status.sh jiminc77 {CODE_REPO} research-dashboard
- state:blocked-human 이 있으면: 해당 GATE REQUEST 를 요약해 나에게 보고하고 VERDICT 를 기다린다
- 없으면: MGMT [Milestone] 이슈의 마지막 [CHECKPOINT] 코멘트 기준으로 다음 STEP 을 이어간다

§7 · 게이트 응답법

푸시가 오면 이렇게 답한다

게이트는 세션이 사람을 기다리는 분기점이다. 폰 알림 → 이슈 열기 → GATE REQUEST 읽기 → GATE VERDICT 댓글 순으로 처리한다.

1

이슈 열기

알림 제목의 레포명·이슈 번호로 해당 CODE 레포 이슈를 연다.

2

GATE REQUEST 읽기

### GATE REQUEST 댓글의 question · options · evidence · impact를 확인한다. class가 soft인지 hard인지 본다.

3

### GATE VERDICT 댓글 작성

같은 이슈에 아래 스키마로 회신한다. 회신 후 gjc/오케스트레이터가 blocked-human → ready로 전환하고 집행한다.

GATE VERDICT 스키마 예시

이슈 댓글 · GATE VERDICT 
### GATE VERDICT
id: P1-M3-G1
choice: B
rationale: <왜 이 선택인지>
follow-ups:
- <추가 지시나 후속 작업, 없으면 생략>

soft 게이트

되돌릴 수 있는 결정. deadline이 지나면 default가 자동 채택되며 다음 세션이 집행한다. 그 전까지는 GATE VERDICT를 올려 override 가능(default 무효).

hard 게이트

방향을 가르거나 되돌리기 어려운 결정. 무기한 대기, 자동 채택 없음. 6시간마다 리마인더 푸시가 온다. go/no-go·임계값 변경·데이터 폐기·아키텍처 선택은 반드시 hard.

§8 · 알림 (ntfy)

topic 문자열이 곧 채널

ntfy에서는 topic 문자열이 곧 채널이다. 폰 ntfy 앱에서 그 topic을 구독하는 일은 평생 1회면 된다 — 한 번 구독해 두면 모든 프로젝트의 게이트 알림이 같은 채널로 온다.

현재 topic은 jiminc77-dgcc-gates-ed9518. 새 프로젝트에서 할 일은 딱 한 줄, 같은 topic을 재사용하도록 secret만 등록하면 된다 (알림 제목에 레포명이 붙어 프로젝트가 구분된다):

shell · 새 프로젝트에서 topic 재사용 
gh secret set NTFY_TOPIC -R jiminc77/{CODE_REPO}

알림이 오는지 확인하는 두 가지 방법

방법 1 · 웹 발행

https://ntfy.sh/ 웹에서 해당 topic으로 테스트 메시지를 발행한다. 폰이 울리면 정상이다.

방법 2 · 수동 실행

CODE 레포 Actions 탭에서 gate-notify를 workflow_dispatch로 수동 실행한다. 대기 게이트가 있으면 리마인더 푸시가 도착한다.

topic 분리를 원하면 폰에서 새 topic을 구독하고 그 프로젝트의 NTFY_TOPIC secret만 바꾸면 된다.

§9 · FAQ

막힐 때 먼저 보는 곳

state 라벨이 2개 붙었어요

단일 state 규칙 위반이다. 사람이 정리해 정확히 1개만 남긴다. 그동안 대시보드는 해당 이슈를 이상으로 표시한다.

CI가 ❌ VERIFICATION FAILED로 떠요

EVIDENCE의 SHA 재검증이 실패한 것이다. 라벨이 state:blocked-tech로 바뀐다. gjc가 블로커를 해소한 뒤 재개한다 (blocked-tech → running).

결과를 본 뒤에 판정 기준을 바꾸고 싶어요

금지가 아니라 절차화한다. MGMT [Decision] 이슈로만 — 무엇을·왜·기존 기준 대비 어떻게 바꾸는지 기록하고 판정한다. 이슈 본문/댓글에서 임의로 재정의하지 않는다.

대시보드에 cached 배지가 떠요

GitHub API rate limit에 걸려 캐시를 표시하는 중이다. 잠시 후 새로고침하면 최신 데이터로 돌아온다. 데이터가 사라진 게 아니라 마지막 성공 응답을 보여주는 것이다.

P{k}.md와 이슈 본문 내용이 달라요

P{k}.md가 canonical이다. 이슈는 명세에서 파생된 작업 단위일 뿐이므로, 충돌하면 P{k}.md를 기준으로 삼는다.

게이트를 놓쳐 세션이 멈춰 있어요

hard 게이트는 무기한 대기하므로 데이터는 안전하다. 이슈를 열어 ### GATE VERDICT를 올리면 즉시 재개된다. 재개 프롬프트(케이스 C)로 상태를 먼저 복원해도 된다.
