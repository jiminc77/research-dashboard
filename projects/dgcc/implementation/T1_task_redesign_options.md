# T1 과제 재설계 옵션 문서 (M3R 게이트 대비 초안)

작성일: 2026-07-07 · 상태: **게이트 대비 자료 — M3R 결과 도착 전 사전 작성** (결정은 게이트에서 데이터와 함께)
전제 데이터: O1 oracle 기준선 (t1a 17–19%, t1b 0–1%, t1c 0%; **min-D 0.13–0.17 vs ε_succ=0.05**; realism ON/OFF 차 ~2%p → 노이즈는 주 원인 아님), M3 정책 (t1a 9%, t1b/t1c ~0%).

## 0. 판정에 사용할 M3R 데이터 (도착 예정)

| 신호 | 무엇을 말해주나 |
|---|---|
| P-c: 굽은-init(u_bend/s_curve) 성공률·min-D 분포 변화 | F1(시간초과 버그) 수리가 다단계 과제를 얼마나 풀어주나 |
| d_at_done·에피소드 내 min-D 분포 (측정 버그 수정 후) | 정책이 실제로 threshold에 얼마나 접근하는가 (기존 final_d는 오염이었음) |
| O2: lift(들기/끌기) 실행 분포 | 정책이 행동 어휘 절반을 쓰고 있는가 (lift head 리스크 확인) |
| O3: flip flicker 발생률 | goal 특징 불연속이 실제 문제인가 |
| (조건부) lookahead oracle | t1b/t1c의 달성 가능성 — 판정 불능 해소 |

## 1. 옵션 목록

### A. Horizon 연장 (T=10 → 20)

- 근거: oracle조차 min-D 0.13에서 정체 — "수가 모자란다" 가설. 10수 × 거친 동작으로는 접근 후 정밀 마무리 여유가 없음.
- 비용: 에피소드 wall 2배. γ=0.95 기준 유효 지평 확인: 0.95²⁰ ≈ 0.36 — 학습 가능 범위.
- 리스크: credit assignment 부담 증가. 성공 신호가 없으면 (C/D 없이는) 연장만으로 부족할 수 있음.
- 파급: 명세 개정 (T는 계획 값). random/oracle 기준선 재측정 필요.

### B. 정밀 마무리 동작 추가 (coarse/fine 2-모드)

- 근거: ‖Δ‖≤0.15 단일 스케일은 "대충 접근"엔 맞지만 최종 5cm 진입에는 과대. lift처럼 scale ∈ {coarse 0.15, fine 0.04} 모드 채널 추가.
- 비용: action space 변경 = **backbone 변경** → P3 공정성 위해 모든 variant 동일 적용, 명세 개정 필요. lift head와 같은 이산-채널 학습 리스크 재발 가능 (동일한 주의 필요).
- 리스크: 탐험 공간 증가. B 단독으로는 성공 신호 부재(닭-달걀)를 못 풂.

### C. 성공 보너스 커리큘럼 (training-only ε-anneal)

- 근거: R_succ=5가 한 번도 발화하지 않음 → 마무리 기술을 배울 신호 자체가 없음. **학습 중** 보너스 트리거를 0.15→0.05로 서서히 조임 (평가·성공 판정은 0.05 고정 — 불변 tier 침범 없음).
- 비용: 구현 반나절. reward 구조 변경이므로 명세 개정 필요 (상수 조정 tier가 아님).
- 리스크: anneal 스케줄이 새 하이퍼파라미터. 트리거 완화 구간에서 Q 스케일 변동.

### D. HER (Hindsight Experience Replay)

- 근거: 실패 에피소드를 "도달한 형상이 goal이었다면 성공"으로 relabel → 성공 신호를 대량 제조. goal-conditioned 구조라 적용이 깨끗하고, **P1.md에 이미 조건부 승인 절차가 존재** (M4 조항 준용).
- 비용: 구현 ~1일 (relabel 시 reward·c_g를 P0 §8 metric으로 재계산 — 정의 이미 문서화됨).
- 리스크: replay 분포 변화 → 안정성 재검 필요 (단 backbone v2 안정화가 선행돼 있음).

### E. Goal 난이도 계층화 (goal curriculum)

- 근거: t1b에서 곡률 작은 goal부터 점진 확대. 탐험이 성공 영역을 밟게 만듦.
- 비용: goal 생성기 파라미터화 (train 분포만; eval 분포 불변). 명세 개정 소요는 경미.
- 리스크: 커리큘럼 스케줄 하이퍼파라미터. eval 분포와의 gap 관리.

### F. ε_succ 재상정 (0.05 → 0.075 등) — 최후수단

- P0 sign-off가 "성공률 천장 관찰 시 재상정 가능"으로 열어둔 문. 단 **연구 방침: 극복 대상이지 완화 대상이 아님** (사용자 확인, 2026-07-06). O1 min-D(0.13)와 실행 노이즈 바닥(median 0.0315·L)을 함께 놓고 보면 0.05는 물리적으로 불가능한 수준은 아님 — 우선순위 최하위 유지.

## 2. 권장 결정 트리 (게이트에서 M3R 데이터 대입)

```
P-c 결과 확인 (F1 수리 효과):
├─ (i) 굽은-init min-D가 0.05 근처까지 하강, 성공 발생
│    → 재설계 불요. 필요시 D(HER)만 소폭 — 사실상 현행 유지
├─ (ii) min-D 0.06–0.12 접근 but 성공 희소 (신호 문제 양상)
│    → C + D 우선 (과제 불변, 저비용) · A는 보류
│    → 파일럿: t1b 1 task × 2 seed × 고속 n_envs, 사전 등록: "min-D 중앙값 < 0.06
│       and 성공 > 10% at 1e5 tr"
├─ (iii) min-D 0.13+ 정체 (oracle과 동급 — 구조 문제 양상)
│    → lookahead oracle 즉시 실행 (달성 가능성 확정용, 고속 레짐)
│    ├─ lookahead 성공 → 학습 문제 확정 → D + E (탐험·신호 강화)
│    └─ lookahead도 0.1+ 정체 → A (+B 검토) 구조 개편, F는 그 후에도 안 되면
└─ 안정성 실패 (halt 재발) 시 → 과제 논의 전면 보류, backbone 재진단 우선
```

## 3. 공통 파일럿 프로토콜 (어느 옵션이든)

- 고속 레짐: 프로브 확장 결과의 knee n_envs 사용 (파일럿은 비교 부채 없음)
- 규모: 1 task × 2 seeds × 1e5 tr (~수 시간), random+O1 기준선 재측정 포함
- 사전 등록: 옵션별 성공 기준을 실행 전 이슈에 고정, 미달 시 다음 분기로
- 변경 격리: 옵션 1개씩 (조합은 단독 효과 확인 후), backbone v2·불변 tier 유지
- 채택 시: P1.md 개정 절차 (G2 §8 선례) + P3 variants 동일 적용 명시
