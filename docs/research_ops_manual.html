DGCC 연구를 GitHub Issues / Projects로 관리하기 위한 최소 운영 기준. Project에는 milestone만 올리고, 문서는 docs에, 구현 증거는 별도 code repo에 둔다.

01 · Final Decision

최종 구조

1. Repo는 2개

research-dashboard는 연구 관리. dgcc는 구현 코드.

2. Project는 milestone만

Task, step, bugfix는 Project에 올리지 않음.

3. Issue는 상태판

긴 plan 복붙 금지. 문서 link, checklist, next, decision만.

핵심: 문서는 지식, Issue는 상태, Project는 흐름. 이 셋을 섞지 않음.

Example 3 · Decision

Title:
[Decision] Use minimal rope_response_probe for GNG-0

## Context

Benchmark tasks add object/task complexity that can confound contact-induced rope response analysis.

## Decision

Use a new minimal `rope_response_probe` task for Go/No-Go 0.

## Evidence

- Plan: `docs/implementation/gng/dgcc_gng_implementation_plan_v2.md`
- Related milestone: `[Milestone] Go/No-Go 0 — Rope Response Probe`

## Why

The Go/No-Go test should isolate contact action → rope shape response.

Benchmark tasks such as coiling, wrapping, lifting, or obstacle-based tasks introduce task-object effects that are not necessary for the first critic latent probe.

## Impact

- Do not use DLO-Lab benchmark tasks for GNG-0.
- Create one minimal task.
- Reward trains a nontrivial Bellman critic but must not equal the probe target.
- Probe target remains contact-induced deformation response.

## Next

Complete Phase A task specification.

07 · Rules

운영 규칙

Project에 넣기

1
Milestone IssueBlueprint, GNG-0, GNG-1, OOD, Paper Draft.
2
현재 milestone 1개Current column에는 하나만 유지.
3
Done 전 link 확인Doc, report, evidence가 연결된 뒤 close.

Project에 넣지 않기

A
세부 stepA0, A1, file 생성, script 수정.
B
작은 bugfixcode repo issue/PR에서 처리.
C
Weekly updateArchive 용도. Milestone issue에 link만.

Weekly update는 세 줄만: State / Blocker / Next. 나머지는 docs와 code repo에 남김.
